# Demo 00 — спайк A2A-канала на пиннутом стеке

Весь прогон доказывает одно утверждение: **на зафиксированных версиях
(`google-adk 2.5.0` + `a2a-sdk[http-server] 1.1.2`) два независимых процесса
разговаривают по протоколу A2A через карту** — без импорта чужого кода и без
единой рукописной строчки транспорта. Это фундамент, на который итерации 1–4
поставят travel-домен; если этот прогон зелёный, «страшная» половина проекта
(межпроцессная граница) уже работает.

Все команды выполняются из корня репо: `/Users/dd/projects/pet/travelwise-lite`.
Перед стартом нужен `.env` (одноразово): `cp .env.example .env`. Для шагов 4–5
в `.env` должен лежать настоящий бесплатный ключ — получить на
https://aistudio.google.com/apikey и заменить им значение `GOOGLE_API_KEY=`.

## 1. Смок-тесты без модели

**Что этим доказываем:** пины реально устанавливаются и работают; ни один тест
не зовёт Gemini и сеть — «зелёное CI» здесь ничего не стоит в RPM.

```bash
make check
```

Ожидаемо: `5 passed` (предупреждения `[EXPERIMENTAL]` от A2A-поверхности ADK —
норма, см. learnings).

## 2. Поднять remote-агента как A2A-сервис

**Что этим доказываем:** агент публикуется как сервис одной строчкой `to_a2a()`
— никакого рукописного FastAPI.

Терминал А:

```bash
make spike-remote
```

Ожидаемо: uvicorn поднялся на `:8001` —

```
INFO:     Uvicorn running on http://localhost:8001 (Press CTRL+C to quit)
```

## 3. Карта на well-known-пути

**Что этим доказываем:** карта — это и есть контракт между владельцами;
она сгенерирована фреймворком и доступна любому клиенту по фиксированному URL,
как OpenAPI-спека.

Терминал Б:

```bash
curl -s http://localhost:8001/.well-known/agent-card.json | jq '{name, description, skills: [.skills[].id]}'
```

Ожидаемо (сам артефакт, не только код ответа):

```json
{
  "name": "hello_supplier",
  "description": "Spike hello agent, reachable over A2A.",
  "skills": ["hello_supplier"]
}
```

## 4. Протокольный хоп ⚠️ live, жжёт free-tier RPM (1 вызов Gemini Flash)

**Что этим доказываем:** граница протокольная, а не импорт — caller знает только
`SUPPLIER_CARD_URL` из env, а ответ remote-агента возвращается через JSON-RPC.

Терминал Б:

```bash
set -a; . ./.env; set +a; echo "hi" | uv run adk run spike/caller
```

Ожидаемо: ответ remote-агента, пришедший через границу —

```
[hello_remote]: Hello there!
```

(текст приветствия может слегка отличаться — это живой Flash).

## 5. Доказательство хопа в логе сервиса

**Что этим доказываем:** вызов действительно прошёл по сети в другой процесс —
проверяем в логе supplier-стороны, не в UI (правило CLAUDE.md: verify in the
log).

Терминал А (лог `make spike-remote`), ожидаемые две строки:

```
INFO:     ::1:NNNNN - "GET /.well-known/agent-card.json HTTP/1.1" 200 OK
INFO:     ::1:NNNNN - "POST / HTTP/1.1" 200 OK
```

`GET` — caller резолвит карту, `POST` — сам JSON-RPC-вызов. Это и есть хоп.

## 6. Оба агента дискаверятся в adk web

**Что этим доказываем:** спайк живёт в конвенции ADK (пакет + `root_agent`),
поэтому встроенный тулинг видит агентов без единой строчки UI-кода.

Терминал Б (сервис из шага 2 должен продолжать работать):

```bash
make web
```

В третьем терминале (или после Ctrl+C по проверке):

```bash
curl -s http://localhost:8000/list-apps
```

Ожидаемо:

```json
["caller","remote_hello"]
```

После проверки погасить оба процесса (Ctrl+C в терминалах А и Б).

## Идемпотентность

Повторный `make check` — тот же `5 passed`; повторный `curl` карты — тот же
JSON; повторный запуск `make spike-remote` при уже занятом `:8001` честно падает
с `[Errno 48] address already in use` — это ожидаемое поведение, а не глюк
(двух supplier-ов на одном порту быть не должно).
