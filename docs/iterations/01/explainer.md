# Итерация 01 — supplier standalone: `ParallelAgent` + детерминированный merge

> 🎯 **Топология — продукт, приложение — фикстура.** Эта итерация построила
> первый настоящий доменный агент — поставщика — и внутри него показала
> дифференциатор №2: встроенную иерархию оркестраторов ADK. Наружу (по сети)
> поставщик пока не смотрит — это iter 2.

## Зачем это (продукт и ценность)

travelwise-lite — планировщик поездок: пользователь пишет «4 days in Lisbon in
May», планировщик разбирает фразу, запрашивает офферы у независимого поставщика
перелётов и отелей и собирает готовый план. Граница между ними — продуктовая:
планировщик и поставщик принадлежат разным владельцам и общаются только по
протоколу, никто не импортирует чужой код. Ценность именно этой итерации — сам
поставщик как работающий продукт: по одной фразе («Lisbon in May, 4 nights») он
ищет перелёты и отели **одновременно**, а не по очереди, и возвращает не
свободный текст, а проверенный структурированный оффер — пакет, который второй
стороне можно машинно читать, не гадая, что там внутри. Пока поставщик отвечает
только локально; сетевой сервис из него сделает следующая итерация.

## 🧵 Что это дало резюме

Дифференциатор №2 — **in-agent иерархия**: внутри одного агента живёт дерево из
встроенных оркестраторов ADK (последовательная обёртка, внутри — параллельная
развилка на двух суб-агентов, следом — детерминированный шаг слияния на чистом
коде). Артефакты-доказательства: живой прогон `adk run supplier` печатает ответы
обоих веток развилки и финальную строку `[merge]` с валидным `SupplierOffer`, а
тест `test_supplier_agent_wiring` фиксирует само дерево. Термины — в глоссарии
ниже.

## TL;DR (простыми словами)

Было: пустой travel-домен — только спайк из пары hello-агентов, доказавший
A2A-канал. Стало: настоящий агент-поставщик из четырёх модулей (схемы, стаб-тулы,
дерево агентов, merge), который на фразу про поездку возвращает проверенный JSON
с перелётами и отелями. Добавились два кусочка: **декларативная параллельная
развилка** (два суб-агента ищут перелёты и отели одновременно, каждый пишет
результат в свой ключ общего состояния) и **шаг слияния на чистом Python-коде**
(без LLM): он читает оба ключа, валидирует и склеивает один оффер. Плюс семь
тестов, которым не нужны ни сеть, ни модель.

## Глоссарий (если ADK и A2A видите впервые)

Базовые термины (ADK, `LlmAgent`, `root_agent`, `adk run`/`adk web`, A2A, agent
card) — см. глоссарий 00. Новое в этой итерации:

- **Оркестратор (workflow-агент)** — агент ADK, который сам не зовёт LLM, а
  управляет запуском своих детей (`sub_agents`). Дерево из таких агентов — это
  то, что в LangGraph вы бы рисовали узлами и рёбрами графа.
- **`SequentialAgent`** — оркестратор «по очереди»: запускает `sub_agents` один
  за другим, каждый видит состояние, оставленное предыдущими. Аналог линейного
  LangGraph-графа, где рёбра проведены цепочкой, — только без самих рёбер.
- **`ParallelAgent`** — оркестратор «одновременно»: запускает всех детей
  конкурентно. Аналог fan-out-ребра в LangGraph, где из одного узла выходят
  ветки в несколько узлов сразу.
- **Session state (состояние сессии)** — общий словарь «ключ → значение», живущий
  в сессии диалога; через него шаги дерева передают друг другу данные. Прямой
  родственник state в LangGraph, но без типизированной схемы: просто строковые
  ключи.
- **`output_key`** — опция `LlmAgent`: финальный текстовый ответ агента
  автоматически сохраняется в state под этим ключом. В LangGraph это была бы
  запись значения в канал state из ноды; здесь — одна строка в декларации.
- **Tool (инструмент, `FunctionTool`)** — обычная Python-функция, отданная
  агенту в `tools=[...]`. ADK сам строит для модели описание вызова из сигнатуры
  и докстринга функции — докстринг и есть контракт, который видит LLM. Аналог
  `@tool`-декоратора в LangChain/CrewAI, только без декоратора.
- **`BaseAgent`** — базовый класс, от которого наследуют кастомный шаг дерева,
  когда нужен не LLM, а свой код. Наследник реализует `_run_async_impl` и
  становится полноправным ребёнком любого оркестратора.
- **Event / `InvocationContext`** — «валюта» рантайма ADK: каждый шаг агента
  отдаёт поток событий (Event — кусок ответа с автором и содержимым), а контекст
  вызова даёт доступ к сессии и её state. Кастомному `BaseAgent`-у приходится
  говорить на этом языке (в LangGraph узел — просто функция над state).
- **`output_schema`** — опция `LlmAgent`, заставляющая модель отвечать строго по
  Pydantic-схеме. Упоминается здесь из-за ограничения: в `LlmAgent` она
  **несовместима с `tools`** — агент с инструментами не может одновременно
  обещать структурный ответ. Это ограничение сформировало дизайн итерации.

## Что это за техника

**`ParallelAgent` — фан-аут декларацией.** Проблема: перелёты и отели — два
независимых поиска, и заставлять их ждать друг друга незачем. В LangGraph это
означало бы: объявить две ноды, провести из одной точки два ребра и — самое
неприятное — написать reducer, разруливающий конкурентные записи в общий state.
В ADK развилка — это буквально `ParallelAgent(sub_agents=[flight, hotel])`, а
конфликт записей снят конвенцией: каждая ветка пишет в **свой** ключ state
(`flights` / `hotels`) через `output_key`, пересечься им негде.

**Merge как кастомный `BaseAgent` — «reducer», вынесенный в следующий шаг.**
Проблема: результаты веток надо слить в один проверенный оффер, и делать это
LLM-ом — значит жечь лишний вызов из бюджета бесплатного тарифа и получать
невоспроизводимый результат. Поэтому слияние — обычный Python-код, оформленный
как `BaseAgent`: он читает оба ключа state, валидирует содержимое Pydantic-схемами
и отдаёт один `SupplierOffer`. В LangGraph такой шаг был бы просто функцией-нодой;
в ADK за место в дереве агентов платишь церемонией событий (см. «Специфика ADK»).

**`SequentialAgent` — скрепка «сначала развилка, потом слияние».** Проблема:
merge обязан стартовать только когда обе ветки дописали свои ключи. Вместо рёбер
и условий — декларация порядка: `SequentialAgent(sub_agents=[fan_out, merge])`.
Гарантия «оба ключа на месте» следует из самой структуры дерева.

**Стаб-тулы как контракт для модели.** Проблема: модель должна вытащить из фразы
пользователя параметры поиска и позвать инструмент правильно. Контракт вызова
ADK строит из сигнатуры и докстринга функции — поэтому `search_flights(destination,
month, nights)` с содержательным докстрингом это и есть «API-спека» для LLM, а
инструкция агенту остаётся поведенческой («позови ровно один раз, верни только
JSON»). Сами тулы — фикстуры без I/O: топология — продукт, данные — декорация.

## ⚖️ Специфика ADK (vs LangGraph/CrewAI)

| Задача | LangGraph/CrewAI | ADK |
|---|---|---|
| Параллельная развилка | Ноды + рёбра из общей точки + reducer для конкурентных записей в state | `ParallelAgent(sub_agents=[...])`; изоляция — разные `output_key`, reducer не нужен |
| Последовательность шагов | Рёбра между нодами, у каждого перехода — своя строчка | `SequentialAgent(sub_agents=[...])` — порядок и есть декларация |
| Детерминированный шаг среди LLM-шагов | Любая нода — просто функция над state, ноль церемоний | Наследник `BaseAgent` с `_run_async_impl`, обязанный эмитить Event с автором и `types.Content` |
| Дать модели инструмент | `@tool`-декоратор, отдельная схема аргументов | Голая функция в `tools=[...]`; декларацию ADK выводит из сигнатуры + докстринга |
| Передать результат ветки дальше | Нода возвращает структуру, она мержится в типизированный state | `output_key` кладёт в state **текст финального ответа модели** — то, что она напечатала |

Честные ложки дёгтя, найденные на этой итерации:

- **`output_key` сохраняет эхо модели, а не результат тула.** Тул вернул список
  словарей — но в state попадает то, что модель *напечатала* по его мотивам:
  ключи может переставить, может обернуть в markdown-заборчик ` ```json `.
  Merge поэтому терпит заборчик (одна регулярка) и валидирует всё Pydantic-ом. В
  LangGraph нода положила бы в state структуру как есть — здесь контроля меньше.
- **`tools` и `output_schema` в одном `LlmAgent` несовместимы.** Заставить
  ветку-агента и позвать тул, и ответить строго по схеме — нельзя; структурность
  пришлось выносить в merge-шаг (валидация постфактум). Это ограничение — одна
  из причин, почему merge вообще существует как отдельный шаг.
- **Кастомный шаг дороже, чем в LangGraph.** «Просто функция» в дереве ADK жить
  не может: подписывайся на `BaseAgent`, генерируй события, заполняй
  `invocation_id`/`author`. Десяток строк церемонии там, где в LangGraph был бы
  `def merge(state): ...`.
- **Фреймворк дрейфует даже внутри пина.** ADK 2.5.0 при каждом конструировании
  дерева предупреждает: `SequentialAgent`/`ParallelAgent` deprecated в пользу
  `Workflow` — при этом сам же признаётся, что `Workflow` «cannot yet be used as
  an LlmAgent sub-agent», то есть замена ещё не готова. Мы остаёмся на пинах
  (см. learnings) — но это наглядная цена vendor-темпа против стабильности
  LangGraph-API.

## Поток данных

Триггер — пользователь пишет фразу в `adk run supplier` (терминальный чат с
агентом). Чтобы собрать оффер, нужны оба списка — перелёты и отели, — и они
независимы: поэтому первым шагом дерево разветвляется. Каждой ветке нужен
источник данных — поэтому у каждой ровно один стаб-тул. Следующему шагу нужны
результаты обеих веток — поэтому ветки пишут в свои ключи state, а merge стоит
*после* развилки в последовательной обёртке и читает оба ключа. Наружу должен
уйти проверенный контракт — поэтому merge не верит тексту модели, а прогоняет
его через Pydantic.

```
пользователь: "Lisbon in May, 4 nights"
      │  adk run supplier
      ▼
SequentialAgent "supplier"
      │ шаг 1: развилка
      ▼
ParallelAgent "fan_out"
  ├─ flight_agent (Flash-Lite) ─ search_flights(...) ─► state["flights"] = "[{...}]"
  └─ hotel_agent  (Flash-Lite) ─ search_hotels(...)  ─► state["hotels"]  = "[{...}]"
      │ шаг 2: оба ключа записаны
      ▼
MergeAgent "merge" (код, не LLM)
  state["flights"] + state["hotels"] ─ парс ─ Pydantic ─► SupplierOffer
      ▼
[merge]: {"flights":[...3 рейса...],"hotels":[...3 отеля...]}
```

| Компонент | Что делает | Что отдаёт |
|---|---|---|
| `supplier/agent.py` | Собирает дерево: две LLM-ветки на Flash-Lite внутри `ParallelAgent`, следом merge, всё в `SequentialAgent` | `root_agent` для `adk run`/`adk web` |
| `supplier/tools.py` | Стаб-поиск: фиксированные варианты по Лиссабону + дефолт для незнакомых городов, ноль I/O | 2–3 словаря-оффера на вызов |
| `supplier/merge.py` | Читает `flights`/`hotels` из state, снимает markdown-заборчик, валидирует схемами | Одно событие с JSON `SupplierOffer` |
| `supplier/schemas.py` | Pydantic-контракт границы: `TripRequest`, `FlightOption`, `HotelOption`, `SupplierOffer` | Валидация в merge и тестах |
| `tests/test_supplier.py` | Семь проверок: схемы, детерминизм стабов, merge (happy + мусор), дерево | Зелёный `make check` без сети |

Честные оговорки — чего в этой итерации **не** происходит: поставщик ещё не
сервис (ни карты, ни протокола — A2A-обвязка это iter 2, planner — iter 3);
физическая одновременность веток строго не доказана (задекларирована деревом,
зафиксирована wiring-тестом, и порядок эхо-строк веток меняется от прогона к
прогону — но настоящий артефакт-трейс покажет `adk web` в iter 4); схема
`TripRequest` в рантайме не используется — модель читает фразу напрямую, а
схема лежит наготове как контракт границы для iter 2–3.

## Карта «где в коде»

1. **Контракт границы — `supplier/schemas.py:11-34`.** Четыре Pydantic-модели,
   ровно по SPEC, без расширений. Докстринг модуля прямо объявляет: это *копия*
   контракта, которую planner в iter 3 продублирует сознательно — общий
   `shared/`-модуль склеил бы обратно те стороны, которые A2A разделяет:

   ```python
   class TripRequest(BaseModel):
       destination: str
       month: str
       nights: int
       travelers: int = 1

   class SupplierOffer(BaseModel):
       flights: list[FlightOption]
       hotels: list[HotelOption]
   ```

2. **Стаб-тулы — `supplier/tools.py:35-45`.** Две функции с одинаковой
   сигнатурой `(destination, month, nights)`; вариативность — только по
   `destination` через словарный lookup с дефолтом, никакой «умной» логики.
   Возвращаются копии словарей, чтобы никто не мутировал модульные фикстуры;
   докстринги написаны как описания для модели — это и есть тул-контракт:

   ```python
   def search_flights(destination: str, month: str, nights: int) -> list[dict]:
       """Search round-trip flight options to the destination for the given month and
       number of nights. Returns a list of flight option objects."""
       # Shallow copies: callers must not be able to mutate the module fixtures.
       return [dict(o) for o in _FLIGHTS.get(destination.strip().lower(), _FLIGHTS[_DEFAULT])]
   ```

3. **Дерево агентов — `supplier/agent.py:25-55`.** Фабрика `_lookup_agent`
   собирает одну ветку развилки (Flash-Lite + один тул + один `output_key`), а
   `root_agent` декларирует всю иерархию одним выражением. Инструкция ветке —
   поведенческая («позови тул ровно раз, верни только JSON»), контракт
   параметров не дублирует — он живёт в сигнатуре тула:

   ```python
   root_agent = SequentialAgent(
       name="supplier",
       sub_agents=[
           ParallelAgent(
               name="fan_out",
               sub_agents=[
                   _lookup_agent("flight", search_flights, "flights"),
                   _lookup_agent("hotel", search_hotels, "hotels"),
               ],
           ),
           MergeAgent(name="merge"),
       ],
   )
   ```

4. **Детерминированный merge — `supplier/merge.py:27-59`.** Чистая функция
   `build_offer` (парс + Pydantic-валидация, тестируется без ADK-рантайма) и
   тонкая ADK-обёртка `MergeAgent`, которая читает state и эмитит одно событие.
   Толерантность к формату ответа модели сознательно закрыта на одной регулярке
   для markdown-заборчика — маркер `# tw-lite:` в модуле фиксирует потолок:

   ```python
   def build_offer(flights_raw: object, hotels_raw: object) -> SupplierOffer:
       """Validate both fan-out results into one SupplierOffer (raises on bad input)."""
       return SupplierOffer(
           flights=_parse_options(flights_raw, "flights"),
           hotels=_parse_options(hotels_raw, "hotels"),
       )

   class MergeAgent(BaseAgent):
       async def _run_async_impl(self, ctx: InvocationContext) -> AsyncGenerator[Event, None]:
           offer = build_offer(ctx.session.state.get("flights"), ctx.session.state.get("hotels"))
           yield Event(..., content=types.Content(role="model",
                       parts=[types.Part(text=offer.model_dump_json())]))
   ```

5. **Ленивый пакет — `supplier/__init__.py:10-15`.** Урок спайка про
   env-на-импорте применён: `agent.py` читает `MODEL_FLASH_LITE` при импорте
   (иначе `adk`-дискавери не увидит модульный `root_agent`), поэтому пакет
   отдаёт его лениво через модульный `__getattr__` (PEP 562) — чистые модули
   `schemas`/`tools`/`merge` импортируются без какого-либо env:

   ```python
   def __getattr__(name: str):
       if name == "root_agent":
           from .agent import root_agent
           return root_agent
       raise AttributeError(name)
   ```

6. **Тесты — `tests/test_supplier.py:15-89`.** Семь проверок без сети и модели:
   дефолты `TripRequest`; стабы валидны по схемам; детерминизм и фолбэк
   незнакомого города; merge собирает оффер из JSON-текстов, терпит заборчик и
   отвергает мусор (нет ключа / не-JSON / не-массив / не та форма); wiring-тест
   через `monkeypatch` env фиксирует всё дерево — классы оркестраторов,
   соответствие тул ↔ `output_key` и модель каждой ветки.

сползти.*
