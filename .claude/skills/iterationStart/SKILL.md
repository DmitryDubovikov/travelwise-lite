# iterationStart

Аргумент — номер итерации N (из `$ARGUMENTS`). Это **полный драйвер итерации**: спека → реализация → ревью → полировка (`/simplify`). Код добивается качества **здесь** — `/iterationClose` потом только церемония без правок кода (доки, ROADMAP, commit-месседж).

## Фазы (по порядку)

### Фаза 0 — Контекст
1. Прочитай строку #N в `ROADMAP.md`: колонку **«Новое»** (какая ОДНА новая ADK-техника вводится), колонку **«🧵 Красная нить»** (какой из трёх дифференциаторов — A2A / in-agent иерархия / встроенный тулинг — итерация делает демонстрируемым), «Скоуп» и «Verify». Сверься с разделом «Заметки» ROADMAP — у итерации может быть заготовленное решение, локально переизобретать нельзя.
2. Прочитай `CLAUDE.md` (правило-зонтик «топология — продукт, приложение — фикстура» + hard constraints: $0/free tier, стабы, никакого своего UI, крошечный домен, пины, «тесты не зовут модель», конвенции ADK главнее дерева, discovery только через `SUPPLIER_CARD_URL`) и релевантные куски `SPEC.md` (форма агентов, model map, контракты, дерево) + `DECISIONS.md` → «Architecture decisions».
3. **Если фундамент поплыл — остановись.** Если спайк iter 0 показал, что ADK-API разошёлся со SPEC (сигнатура `to_a2a()`, well-known-путь карты, поведение `RemoteA2aAgent`), — сначала предложи правку SPEC/DECISIONS и дождись подтверждения, потом пиши спеку. Не подгоняй код под устаревшую спеку и наоборот молча.

### Фаза 1 — Спека
4. Создай `specs/NN/spec.md` (NN двузначный) по шаблону ниже. **Тонко: ≤1 страницы.** Спека обязана процитировать пункт красной нити из ROADMAP и назвать **сиблинг-аналог** новой ADK-техники (через что знакомое она осваивается: `SequentialAgent` ≈ линейный LangGraph-граф, `ParallelAgent` ≈ fan-out-ребро, A2A ≈ граница, которую в LangGraph пришлось бы катать руками поверх HTTP, `adk web` ≈ Chainlit/LangSmith-подобная поверхность «из коробки»). Покажи её словами.

### Фаза 2 — Реализация
5. Реализуй по спеке до выполнения **done-gate** (по факту существования). Соблюдай швы: planner знает supplier **только** через `SUPPLIER_CARD_URL` (env) — никакого импорта supplier-кода и хардкода host/port; схемы — **осознанный дубль** (`planner/schemas.py` / `supplier/schemas.py`), общий модуль не заводить; `merge` — детерминированный код, не LLM; A2A-клиент — только `RemoteA2aAgent`, карты генерит фреймворк (никаких ручных `a2a_client.py`/`card.py`); state-ключи фан-аута — `flights`/`hotels`. Домен не расширять: новое поле/сущность/суб-агент сверх SPEC — это scope-creep, главный failure mode siblings.
6. **AI-дисциплина.** Модели — только конкретные пиннёные id из `.env` (Flash / Flash-Lite, никогда `-latest` и никогда Pro), model map из SPEC не пересматривать. **Тесты и CI не зовут модель и сеть** — pytest покрывает чистый Python (схемы, стабы, merge, карта). Одиночные live-вызовы бесплатны, но жгут free-tier RPM/RPD (~4 вызова на запрос при ~15/~30 RPM) — батч-прогоны и eval-циклы предлагай только явно, с оценкой числа вызовов.

### Фаза 3 — Ревью и полировка (финальные шаги реализации)
Гоняется, когда done-gate функционально достигнут и код субъективно готов (не по полу-написанному). Цель — **качество кода, НЕ артефакты**: находки живут только в контексте, применяются, испаряются. Никаких `docs/reviews/`.

7. **Два ревьюера параллельно** (в одном сообщении, два вызова Agent — свежий контекст):
   - `general-reviewer` — логика/edge-cases/простота/scope-creep;
   - `constitution-reviewer` — швы `CLAUDE.md` + консистентность с аналогами.
   Каждому передай путь `specs/NN/spec.md`. Дифф они собирают сами с рабочего дерева — НЕ передавай его.
8. **Дедуп.** Одна проблема в одном месте под разными углами → склей, severity = максимум, помечай источник.
9. **Аудитор.** Вызови `review-auditor`, вставив склеенный список находок в промпт. Он выносит KEEP / DOWNGRADE→X / DROP / REWORD — страж против кривого авто-фикса по ложному срабатыванию.

   **Результат аудитора забирай pull, не push.** Push-нотификация о завершении агента иногда не доходит (агент отработал, но `<task-notification>` не прилетела — выглядит как «зависание», хотя вердикт готов). Если нотификация не пришла в разумный срок (~60–90с), **не жди дальше** — вытащи финальный ответ прямо из транскрипта агента по пути `output_file`, который вернул вызов Agent. **Только хирургически через `jq`** (не `Read`/`cat`/`tail` целиком — переполнит контекст):
   `jq -r 'select(.message.content) | .message.content[]? | select(.type=="text") | .text' <output_file> | tail -40`
   Если у последней записи `stop_reason=end_turn` — агент завершился штатно, бери этот текст как вердикт. Если транскрипт реально оборвался на pending tool_use (а не на тексте) — вот это настоящий зависший вызов: останови агент (`TaskStop`) и применяй **консервативный фолбэк** (см. шаг 10).
10. **Применить вердикты:** DROP → выбросить; DOWNGRADE → сменить severity; REWORD → поправить якорь. Затем **CRITICAL + BUG → чини**; **NIT** → только если правка очевидно безопасна. После фиксов прогони **точечные** pytest-тесты (без LLM, без сети), чтобы фикс не сломал happy-path.
    - **Консервативный фолбэк (аудитор реально недоступен).** Если вердикт не достать ни push, ни pull: выступи аудитором сам по его же критериям, но строго в защиту кода — по умолчанию KEEP только то, что чинишь **очевидно безопасно** (NIT-косметика); на спорное/CRITICAL без явного подтверждения авто-фикс **не накладывай**. Лучше оставить находку на ревью пользователю, чем испортить корректный код. Отметь в хэндоффе, что аудитор был недоступен.
11. Существенные фиксы — можно прогнать ревьюеров повторно. Цель фазы: CRITICAL/BUG = 0.
12. **Причесать код.** Когда ревью чисто, вызови встроенный навык `/simplify` по текущему диффу — косметические правки качества поверх результата ревью, включая код самих фиксов. Это не охота на баги — корректность и швы уже отловлены. После — точечные pytest-тесты ещё раз.

### Фаза 4 — Хэндофф
13. Короткая сводка: done-gate выполнен; **какой дифференциатор стал демонстрируемым**; ревью — 2-4 строки (что починил по severity, что аудитор отбросил); что причесал `/simplify`; осознанные срезы (`# tw-lite:`-маркеры), если решили что-то НЕ чинить (пойдут в `learnings.md` при close).
    - **Итерация ещё НЕ закрыта.** ROADMAP остаётся ⬜, доки не сгенерированы, изменения не застейджены, пока не прогнан `/iterationClose N`. Явно скажи это и предложи следующий шаг — `/iterationClose N`.

```markdown

## 🧵 Красная нить
<дословно из ROADMAP: какой дифференциатор (A2A / иерархия / тулинг) итерация делает демонстрируемым>

<через что знакомое осваиваем: SequentialAgent ≈ линейный LangGraph-граф, ParallelAgent ≈ fan-out,

(3-5 пунктов; последний — ревью-пайплайн)
```

## Правила
- Спека тонкая (≤1 страницы). Детали оркестрации ревью живут здесь, в скилле, не в спеке.
- Батч-прогоны/eval-циклы с live Gemini — только с явной пометкой про расход RPM/RPD и со спроса.
- Развилки стека уже решены (`DECISIONS.md` → «Architecture decisions»): RemoteA2aAgent, дубль схем, детерминированный merge, model map — не пересматривай. Существенные новые развилки — спрашивай.
- **Разрешения на рутинные команды** (`make`/`uv`/read-only `git`/`curl localhost`) преднастроены в `.claude/settings.json` — не спрашивай подтверждения повторно. Новая рутинная команда того же рода регулярно триггерит подтверждение → добавь точечный паттерн в `.claude/settings.json` (не в `settings.local.json` — личный файл). Push/force/`rm -rf` в allow-лист не тащи — они должны спрашиваться каждый раз.
